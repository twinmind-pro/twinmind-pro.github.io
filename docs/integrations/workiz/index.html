Workiz Integration Guide

1. What This Integration Does​

The Newo AI Agent integrates with Workiz so your AI can create and cancel jobs inside Workiz during a live conversation.

Once connected, the agent can:

  • create a new Workiz job from the booking payload collected in conversation
  • cancel an existing Workiz job when the booking contains a Workiz booking ID
  • recognize returning callers by phone number and load their upcoming jobs into persona bookings
  • sync dispatcher-capable team members from Workiz and use one of them as the default assignee
  • automatically add booking and cancellation scenarios to Canvas
  • run in DevMode and emulate Workiz API responses without touching production data

Important limitation: this integration does not provide live availability checking. In setup, the availability tool is explicitly disabled and the main user-facing flows are booking, cancellation, and existing-client lookup.

2. Before You Start​

Make sure you have:

  • an active Workiz account
  • access to Workiz API credentials
  • access to Newo Builder for the target project
  • permission to edit attributes and publish the project
  • at least one dispatcher-capable Workiz team member with role dispatch, manager, or admin
  • a real list of Workiz job type names

You will need these values:

  • API Key - used in the Workiz request URL path
  • API Secret - sent in Workiz POST request bodies
  • Job Types - a JSON array with exact Workiz job type names
  • Primary Assigned User - selected after the first successful publish and team sync

Example Job Types value:

["HVAC Repair", "Plumbing", "Electrical"]

Use the exact names from Workiz. The booking flow validates jobType against this list.

3. Connecting for the First Time​

![Workiz module selection](/img/integrations/workiz/select module.png)

Figure 1. Select workiz_integration in the project module creation form.

3.1 Fill in the Core Settings​

  1. Open your project in Newo.
  2. Go to Builder -> Attributes.
  3. Find the Workiz section.
  4. Set Execution Mode:
    • DevMode for safe testing without real API changes
    • Production for real Workiz requests
  5. Fill in API Key.
  6. Fill in API Secret.
  7. Leave Base URL as https://api.workiz.com/api/v1 unless your team gave you a different endpoint.
  8. Set Enable Booking to True if the AI should create Workiz jobs.
  9. Set Enable Cancellation to True if the AI should cancel Workiz jobs.
  10. Set Automatically Add Default Scenarios to Canvas to True if you want Newo to prepare Canvas items automatically.
  11. Fill in Job Types with a JSON array of exact Workiz job type names.
  12. Click Save.
  13. Click Publish All.

Figure 2. Publish the project after filling Workiz settings.

3.2 What Happens After Publish​

After publish, Newo runs Workiz setup logic:

  • the HTTP connector is prepared
  • booking and cancellation tools are enabled for ConvoAgent
  • the payload schema for booking is registered
  • if API Key is filled, Newo requests /team/all/
  • active dispatcher-capable team members are cached
  • the Primary Assigned User dropdown is refreshed from synced team members

If API Key is empty, the team sync is skipped.

3.3 Select the Default Assignee​

After the first successful publish:

  1. Re-open the Workiz attribute block if needed.
  2. Find Primary Assigned User.
  3. Select the team member who should be assigned to newly created jobs by default.
  4. Save and publish again if you changed the value.

5. What the AI Agent Can Do​

5.1 Create a Workiz Job​

The booking flow accepts a payload from ConvoAgent and creates a Workiz job through /job/create/.

What happens automatically:

  • validates required fields such as bookingDateTime, jobType, customerInfo.firstName, and customerInfo.phoneNumber
  • normalizes the phone number
  • sends the create request to Workiz
  • assigns the selected Primary Assigned User through /job/assign/
  • returns a success result back to ConvoAgent

5.2 Cancel a Workiz Job​

The cancellation flow updates an existing Workiz job through /job/update/.

What happens automatically:

  • reads payload.meta.booking_id
  • validates that API Secret and booking ID are present
  • sends a cancellation update with Status = Canceled
  • returns a success result back to ConvoAgent

5.3 Recognize Returning Clients​

When the system knows the caller phone number, ExistingClientFlow requests /job/all/ and filters results by phone number.

Only future jobs are saved into the persona bookings attribute. This allows the AI to:

  • know whether the caller already has a booking
  • present existing bookings during cancellation flows
  • keep Workiz booking metadata such as booking_id, status, and job_type

5.4 Prepare Canvas Automatically​

When automatic Canvas setup is enabled, Workiz adds:

  • booking scenario workiz_schedule_appointment_via_agent_scenario
  • cancellation scenario cancellation_via_agent_scenario
  • cancellation intent cancellation_via_agent_intent

During setup, Newo also removes generic scheduling and cancellation items that would conflict with the Workiz-specific flow.

5.5 Run Safe Tests in DevMode​

When Execution Mode = DevMode, the integration does not call the real Workiz API.

Instead, it emulates responses for:

  • /team/all/
  • /job/all/
  • /job/create/
  • /job/assign/
  • /job/update/

This is the safest way to validate setup logic, Canvas wiring, booking flow behavior, and cancellation flow behavior before moving to production.

5.6 What Is Not Supported Yet​

The current Workiz integration does not support:

  • live availability lookup
  • rescheduling as a dedicated flow
  • conversation sync back into Workiz CRM
  • pipeline or lead-stage automation

6. How to Test That Everything Works​

6.1 After Setup​

After clicking Publish All, confirm:

  • Publish finishes without setup errors
  • Primary Assigned User is populated from dispatcher-capable team members
  • Dispatcher Team Members contains synced values
  • Booking and cancellation tools are enabled in the project

6.2 Test in DevMode First​

  1. Set Execution Mode to DevMode.
  2. Publish the project.
  3. Start a test conversation.
  4. Ask the AI to book a job.
  5. Confirm the agent completes the booking flow successfully.
  6. Ask the AI to cancel the same booking.
  7. Confirm the cancellation flow completes successfully.

Expected result:

  • no real Workiz data is changed
  • booking and cancellation success events still return through ConvoAgent

6.3 Test Real Booking in Production​

  1. Set Execution Mode to Production.
  2. Publish the project.
  3. Start a test conversation.
  4. Provide:
    • first name
    • last name
    • phone number
    • email
    • service address
    • booking date and time
    • valid job type
  5. Let the AI submit the booking.
  6. Check Workiz.

Expected result:

  • a new job is created in Workiz
  • the selected primary assignee is attached to the job
  • JobSource is filled with the provided value or the default job source

6.4 Test Existing Client Recognition​

  1. Use the same phone number from a future Workiz job.
  2. Start a new session.
  3. Confirm that the integration loads existing jobs for that phone number.

Expected result:

  • future jobs are stored in the persona bookings attribute
  • cancellation flow can use those jobs as context

6.5 Test Cancellation​

  1. Use a booking that contains a valid Workiz booking_id.
  2. Ask the AI to cancel the booking.
  3. Check Workiz.

Expected result:

  • the job status becomes Canceled
  • ConvoAgent receives result_success for cancel_booking

6.6 Test Canvas Setup​

  1. Keep Automatically Add Default Scenarios to Canvas enabled.
  2. Publish the project.
  3. Open Canvas and Library.

Expected result:

  • Workiz booking scenario exists
  • Workiz cancellation scenario exists
  • cancellation intent exists
  • conflicting generic scheduling and cancellation items are removed

Figure 4. Canvas after setup, with Workiz booking and cancellation scenarios added.

7. Troubleshooting​

Booking does not start​

Check:

  • Enable Booking is True
  • the booking payload includes required fields
  • Job Types contains the exact requested value

Booking fails after submission​

Most common causes:

  • API Secret is empty
  • Primary Assigned User is not selected
  • jobType is not included in workiz_job_types

Cancellation fails​

Check:

  • Enable Cancellation is True
  • the booking contains meta.booking_id
  • API Secret is correct

Primary Assigned User is empty after publish​

Check:

  • API Key is filled
  • /team/all/ can be reached
  • the Workiz users have roles dispatch, manager, or admin

Existing-client lookup returns nothing​

Check:

  • the caller phone number matches Workiz job phone numbers after normalization
  • the job is in the future
  • the Workiz response includes the job in /job/all/
