GoHighLevel Integration Guide

1. Document Purpose​

This document is the operational guide for the current Newo AI Agent integration with GoHighLevel (GHL).

It is written for:

  • partner teams implementing the integration for clients
  • end clients who need to connect, configure, test, and support the integration

This guide covers:

  • how to connect the integration
  • how to reconnect or rotate credentials
  • which scenarios and automations are supported today
  • how to launch and test those automations
  • what the Canvas setup adds automatically
  • known limitations, FAQs, and troubleshooting

This document reflects the current implementation in this repository, not a future roadmap.

Figure S01. Entry point for starting GoHighLevel integration setup in Newo.

2. What This Integration Does​

The GoHighLevel integration connects a Newo AI Agent to GoHighLevel so that the agent can use GHL for:

  • appointment availability lookup
  • appointment booking
  • appointment cancellation
  • appointment rescheduling
  • contact lookup and contact creation
  • existing client detection by phone number
  • conversation synchronization back to GHL
  • contact note/history updates
  • opportunity and pipeline stage updates
  • Canvas scenario setup for booking-related flows

In practical terms, this means the AI agent can talk to a lead or customer, use GHL as the CRM and calendar backend, and write the outcome back into GHL automatically.

4. Connection Options​

The integration supports two connection methods:

  • OAuth
  • Private Token

4.1 OAuth vs Private Token​

MethodRecommendedBest ForStrengthsLimitations
OAuthYesMarketplace-style setup, partner-managed deploymentsCleaner authorization flow, connector-based setup, better fit for packaged app distributionRequires authorization step and code handling
Private TokenConditionalManual setup in a specific GHL sub-accountSimple to understand, direct manual controlScope-dependent; can be more limited operationally than Marketplace/OAuth setups

4.2 Important Note About Private Token​

Private Token setups can be functionally limited depending on:

  • the scopes granted to the token
  • the way the GHL application is installed
  • whether the account expects Marketplace app behavior
  • whether additional communication channels are provided through the Marketplace app setup

In other words:

  • OAuth / Marketplace-style setup is the safer option when the deployment depends on the full app experience
  • Private Token is acceptable for manual setups, but partners should verify channel availability and account entitlements before go-live

5. Prerequisites​

Before setup, confirm the following:

  • active GoHighLevel account
  • correct GHL sub-account / location to connect
  • access to Newo project builder
  • permission to edit project attributes and publish
  • either:
    • OAuth authorization access, or
    • a GHL Private Integration Token with the required scopes

Required operational inputs:

  • ghl_auth_type
  • ghl_location_id
  • one of:
    • ghl_oauth_code for OAuth
    • ghl_oauth_token for Token

6. Quick Start​

Use this section if you want the shortest path to a working integration.

  1. Open the project in Newo.
  2. Go to Builder -> Attributes.
  3. Set ghl_auth_type to either OAuth or Token.
  4. Fill in ghl_location_id.
  5. If using OAuth, paste ghl_oauth_code.
  6. If using Token, paste ghl_oauth_token.
  7. Review the core feature flags:
    • ghl_enable_slot_check
    • ghl_enable_booking
    • ghl_enable_cancellation
    • ghl_sync_conversation
    • ghl_enable_client_history
    • ghl_update_pipeline
    • ghl_setup_scenarios
  8. Click Save.
  9. Click Publish All.
  10. Verify that calendars and pipelines are populated after publish.

If publish succeeds, the setup flow will attempt to:

  • configure the connector
  • fetch calendars
  • fetch custom values
  • fetch pipelines
  • refresh dynamic dropdowns such as default calendar and selected pipeline

7. Initial Connection​

7.1 Connect with OAuth​

Use OAuth when:

  • the integration is installed in a Marketplace-style flow
  • you want the recommended path
  • you want the lowest operational friction for partners and clients

Steps:

  1. In Newo, set ghl_auth_type to OAuth.
  2. Open the authorization link provided in the ghl_oauth_code field description.
  3. Log in to GoHighLevel.
  4. Approve the requested scopes.
  5. Select the correct sub-account / location.
  6. Copy the authorization code returned after redirect.
  7. Paste it into ghl_oauth_code.
  8. Set ghl_location_id.
  9. Click Save.
  10. Click Publish All.

Expected result:

  • connector setup starts
  • calendars are fetched
  • pipelines are fetched
  • the integration becomes ready for runtime events

Figure S03. OAuth authorization screen in GoHighLevel before approval.

Figure S04. Result page where the authorization code can be copied for Newo setup.

7.2 Connect with Private Token​

Use Private Token when:

  • the deployment is manual
  • the client prefers direct token-based setup
  • Marketplace / OAuth is not available for that account

Steps:

  1. In GoHighLevel, open Settings -> Integrations.
  2. Create or open a Private Integration.
  3. Generate a token with the required scopes.
  4. In Newo, set ghl_auth_type to Token.
  5. Paste the token into ghl_oauth_token.
  6. Set ghl_location_id.
  7. Click Save.
  8. Click Publish All.

Important:

  • the token must have all required scopes
  • a token with missing scopes may let setup finish partially but still break runtime actions later

Minimum operational expectation for the token:

  • contacts read/write
  • conversations read/write
  • conversation messages read/write
  • calendars read/write
  • calendar events read/write
  • calendar groups/resources read
  • businesses read/write
  • locations read
  • location custom values/custom fields read/write
  • opportunities read/write
  • users read
  • workflows read

8. Reconnection and Credential Rotation​

This section is critical for support teams.

8.1 When to Reconnect​

Reconnect the integration when:

  • the OAuth authorization is stale or invalid
  • the token has been revoked or rotated
  • calendars or pipelines stop refreshing after publish
  • runtime actions suddenly fail after account-side changes
  • the client changed location/account context

8.2 Reconnect OAuth​

Use this playbook if OAuth is already selected.

  1. Confirm ghl_auth_type = OAuth.
  2. Generate a fresh authorization code.
  3. Replace the old value in ghl_oauth_code.
  4. Reconfirm ghl_location_id.
  5. Click Save.
  6. Click Publish All.
  7. Verify calendar and pipeline dropdowns refresh.
  8. Run an availability test.

Recommended verification after OAuth reconnect:

  • default calendar list appears
  • selected pipeline list appears
  • book_slot succeeds in a test environment

8.3 Rotate a Private Token​

  1. Generate a new GHL Private Integration Token.
  2. Replace the current value in ghl_oauth_token.
  3. Confirm ghl_auth_type = Token.
  4. Confirm ghl_location_id.
  5. Click Save.
  6. Click Publish All.
  7. Run a simple test flow such as availability or existing client lookup.

8.4 Switch from Token to OAuth​

  1. Set ghl_auth_type = OAuth.
  2. Paste a valid ghl_oauth_code.
  3. Keep ghl_location_id aligned with the intended sub-account.
  4. Save and publish.
  5. Re-test the runtime flows.

8.5 Switch from OAuth to Token​

  1. Set ghl_auth_type = Token.
  2. Paste a valid ghl_oauth_token.
  3. Confirm the token scopes.
  4. Save and publish.
  5. Re-test.

8.6 Safe Reconnect Rule​

After any auth-related change, always do all three:

  • Publish All
  • verify dynamic dropdowns refreshed
  • perform at least one live-path test

9.3 Calendar Selection Modes​

single​

  • uses the fixed calendar selected in ghl_calendar
  • best for one-calendar businesses
  • safest mode for production

name​

  • AI extracts the requested service or provider name
  • the integration matches that against available calendars
  • depends heavily on a good ghl_selection_criteria prompt

duration​

  • AI extracts desired appointment length
  • the integration matches it to a suitable calendar
  • useful when calendar differences are primarily based on duration

9.4 Configuration Advice​

Recommended defaults for most production deployments:

  • ghl_auth_type = OAuth
  • ghl_selection_type = single
  • ghl_enable_slot_check = True
  • ghl_enable_booking = True
  • ghl_enable_cancellation = True
  • ghl_check_existing_client = True
  • ghl_sync_conversation = True
  • ghl_enable_client_history = True
  • ghl_update_pipeline = False until rules are tested
  • ghl_setup_scenarios = True

10. Supported Scenarios​

10.1 Availability​

Trigger:

  • get_available_slots
  • optionally session_started

What happens:

  • the integration checks whether availability lookup is enabled
  • chooses a calendar using single, name, or duration
  • requests free slots from GHL
  • returns availability back to the agent

Best for:

  • “What times do you have tomorrow?”
  • “Do you have a 30-minute consultation this week?”

10.2 Booking​

Trigger:

  • book_slot

What happens:

  • validates booking is enabled
  • looks up or creates the contact
  • selects the calendar
  • creates the appointment in GHL

Best for:

  • confirmed scheduling requests

10.3 Cancellation​

Trigger:

  • convo_agent_cancel_booking

What happens:

  • validates cancellation is enabled
  • uses the booking identifier
  • updates appointment status to cancelled

Important:

  • cancellation depends on the agent correctly identifying the booking to cancel
  • the default Canvas content includes a dedicated cancellation intent and cancellation scenario

10.4 Reschedule​

Trigger:

  • reschedule_booking_tool

What happens:

  • extracts the existing booking and target time
  • updates the appointment start time in GHL

10.5 Existing Client Detection​

Trigger:

  • define_user_phone_number

What happens:

  • searches the contact by phone
  • retrieves existing future appointments
  • injects context that can improve the conversation

10.6 Conversation Sync​

Trigger:

  • ghl_session_ended

Controlled by:

  • ghl_sync_conversation

What happens:

  • gets or creates the conversation
  • writes an outbound conversation message with call/session context

10.7 Client History Notes​

Trigger:

  • ghl_session_ended

Controlled by:

  • ghl_enable_client_history

What happens:

  • writes summary/transcript information into the contact note history

10.8 Pipeline Update​

Trigger:

  • ghl_session_ended

Controlled by:

  • ghl_update_pipeline

What happens:

  • AI evaluates the session outcome
  • chooses a stage from the selected pipeline
  • updates the opportunity

10.9 Static Data Refresh​

Trigger:

  • session_started

What happens:

  • refreshes cached calendar data
  • helps keep calendar metadata fresh during runtime

  • library_intent_types_common_appointment_agent

It also removes legacy items:

  • scenario_home_service_appointment_request

11.2 User Meaning of This Feature​

This matters to users because it gives them:

  • ready-made appointment handling building blocks

13. How to Launch and Test Automations​

13.1 Launch Publish-Time Automation​

  1. Change one or more ghl_* attributes.
  2. Click Save.
  3. Click Publish All.
  4. Confirm dynamic dropdowns refresh.

13.2 Launch Availability Automation​

  1. Start a chat or voice session.
  2. Ask for available times.
  3. Confirm the agent triggers slot lookup.
  4. Verify returned times are correct for the intended calendar.

13.3 Launch Booking Automation​

  1. Ask the agent to book a confirmed time.
  2. Provide required customer data.
  3. Verify the appointment appears in GHL.
  4. Verify the contact exists or was created.

13.4 Launch Cancellation Automation​

  1. Use an appointment that already exists.
  2. Ask the agent to cancel it.
  3. Confirm the appointment is updated to cancelled in GHL.

13.5 Launch Reschedule Automation​

  1. Use an existing booking.
  2. Ask the agent to move it.
  3. Confirm the new start time in GHL.

13.6 Launch Existing Client Automation​

  1. Start a session where the caller phone is available.
  2. Verify the integration finds the contact.
  3. Verify future bookings are returned as context.

13.7 Launch Session-End Automation​

  1. Complete a call or chat.
  2. Let the session end normally.
  3. Confirm post-session outputs in GHL:
    • conversation message
    • contact note
    • opportunity update if enabled

13.8 Connectivity / Test Endpoints​

  • availability test via test_integration on webhook
  • cancellation test via test_integration on webhook
  • reschedule test via test_integration on webhook
  • existing client test via existing_client_test_event

These are useful for controlled validation, but partner teams should still perform at least one full business-path test in a realistic environment.

15. Troubleshooting​

15.1 Publish Completed but Calendar Dropdown Is Empty​

Possible causes:

  • invalid OAuth code
  • invalid token
  • wrong ghl_location_id
  • calendars are not available in that location
  • publish happened before auth was valid

What to check:

  • ghl_auth_type
  • credential field
  • ghl_location_id
  • re-run Publish All

Fix:

  • reconnect credentials
  • publish again

15.2 Booking Does Not Happen​

Possible causes:

  • ghl_enable_booking = False
  • wrong calendar selected
  • calendar selection prompt is weak
  • missing or malformed customer data
  • contact create/search failed

What to check:

  • feature flags
  • ghl_selection_type
  • ghl_calendar
  • ghl_selection_criteria
  • booking payload completeness

Fix:

  • enable booking
  • use single mode temporarily
  • strengthen prompt instructions
  • test again

15.3 Cancellation Does Not Happen​

Possible causes:

  • ghl_enable_cancellation = False
  • missing booking identifier
  • agent did not resolve the correct appointment
  • GHL request failed

What to check:

  • cancellation flag
  • appointment ID in the cancellation path
  • whether the user actually has a future booking

15.4 Reschedule Does Not Work​

Possible causes:

  • missing original booking ID
  • invalid target time
  • timezone mismatch

What to check:

  • original appointment exists
  • converted datetime is correct
  • business timezone vs user timezone assumptions

15.5 Wrong Calendar Is Used​

Possible causes:

  • ghl_selection_type is name or duration
  • ghl_selection_criteria is ambiguous
  • new calendars were added but metadata was not refreshed

Fix:

  • re-publish
  • simplify selection criteria
  • switch to single for debugging

15.6 No Conversation Data Appears in GHL​

Possible causes:

  • ghl_sync_conversation = False
  • session-end event was not triggered
  • conversation/contact mapping failed

What to check:

  • feature flag
  • session-end logs
  • whether the contact exists and is connected to the session context

15.7 No Note Appears on Contact​

Possible causes:

  • ghl_enable_client_history = False
  • session-end path failed
  • contact resolution failed

What to check:

  • client history flag
  • contact identity
  • session-end logs

15.8 Pipeline Is Not Updating​

Possible causes:

  • ghl_update_pipeline = False
  • no pipeline selected
  • no stages available
  • stage criteria are too weak

What to check:

  • ghl_update_pipeline
  • ghl_pipeline_item
  • ghl_pipeline_stages
  • ghl_pipeline_criteria

15.9 New Calendars or Pipelines Do Not Appear​

Cause:

  • metadata is refreshed on publish/setup, not instantly

Fix:

  • click Publish All

15.10 OAuth Works Poorly or Stops Unexpectedly​

Possible causes:

  • stale authorization
  • wrong location/account during auth
  • connector state drift after account changes

Fix:

  • generate a fresh OAuth code
  • re-save attributes
  • publish again

15.11 Token Setup Works Partially but Some Features Behave Differently​

Cause:

  • token scope limitations or account-level entitlement differences

Fix:

  • verify scopes
  • confirm whether the deployment should use Marketplace/OAuth instead
  • verify whether communication-channel behavior depends on Marketplace app installation

16. Known Limitations and Operational Notes​

These limitations should be communicated early to partners and clients.

  1. One integration instance supports one GHL location at a time.
  2. This integration performs live writes into GoHighLevel.
  3. Historical backfill is not the goal of this implementation.
  4. Calendar selection in name or duration mode depends on prompt quality and available metadata.
  5. Session-start availability preload is only operationally safe when calendar selection is predictable.
  6. Private Token setups may behave differently from Marketplace/OAuth setups depending on scopes and channel entitlements.
  7. Canvas starter scenarios are useful defaults, but they are not a substitute for business-specific conversation design.

17. FAQ​

Does the integration create real records in GoHighLevel?​

Yes. Bookings, cancellations, conversation messages, notes, and opportunity updates affect the live GHL environment.

Can one integration instance handle multiple locations?​

No. Use one integration instance per GHL location.

Which auth method should we prefer?​

Use OAuth unless there is a clear reason to use Private Token.

Can the integration work with a manual Private Token setup?​

Yes, but partner teams should verify scope coverage and account-specific limitations before production launch.

Why do newly created calendars or pipelines not show up immediately?​

Because the integration refreshes that metadata on Publish All.

Can the agent identify existing customers automatically?​

Yes, when the caller phone is available and ghl_check_existing_client is enabled.

Does the integration support rescheduling?​

Yes. The current implementation has a dedicated reschedule flow.

What does ghl_setup_scenarios do?​

It auto-adds default booking/cancellation-related Canvas items for the user, including the cancellation-via-agent content and the speed-to-lead scenario.

Does the integration support post-call CRM updates?​

Yes. Depending on feature flags, it can sync conversation output, write contact history notes, and update pipeline stage.

Can the integration detect timezone from the conversation?​

Yes, if ghl_use_timezone_from_conversation is enabled.

19. GoHighLevel Automations and Native Marketplace Actions​

This section explains two different automation models that are available with the current GoHighLevel integration:

  1. Native Marketplace Actions These are actions exposed inside GoHighLevel that can directly trigger Newo behavior such as outbound calls and SMS.
  2. GHL Workflow Automations Based on Synced Data These are automations built inside GoHighLevel Workflows using data that Newo writes back to GHL after sessions.

Both models are valid and can be used together.

19.1 Native Marketplace Actions Overview​

The Marketplace app provides native GoHighLevel actions that let users trigger Newo tasks directly from GHL workflows and pipelines.

Current native action patterns include:

  • launching an outbound call from GHL into Newo
  • sending SMS from GHL into Newo

This means users can build workflow automations entirely inside GoHighLevel and call Newo as the execution layer.

Typical use cases:

  • call a lead immediately after form submission
  • call a lead when a pipeline stage changes
  • send an SMS after a missed call
  • send an SMS after a lead enters a nurture pipeline
  • trigger callback or reminder actions from GHL workflow logic

Figure S10. GoHighLevel workflow builder with a Newo native action selected inside the automation flow.

19.2 Native Outbound Call Action​

The outbound call action implemented in the Marketplace app follows this pattern:

  1. Read input values from the GHL action.
  2. Validate required fields.
  3. Authenticate in Newo using newo_api_key.
  4. Get or create the voice actor/persona for the phone number.
  5. Ensure there is a webhook/API actor for that persona.
  6. Send an external_request event into Newo with taskTypes = ["make_call"].

Current action inputs used by the app:

  • newo_api_key
  • newo_phone_number
  • newo_contact_name
  • newo_agent_instruction

Current validation behavior:

  • if phone number is empty -> action returns phone number is empty
  • if contact name is empty -> action returns contact name is empty

Current instruction assembly pattern:

  • base instruction tells Newo to call the user at the provided phone number
  • includes the contact name
  • appends optional extra instructions from newo_agent_instruction

Operationally, the native action does the following in Newo:

  • obtains a bearer token using the Newo API key
  • gets or creates the target actor using the phone number
  • resolves the persona
  • finds or creates the webhook/API actor for that persona
  • submits event_idn = external_request
  • sends:
    • content
    • taskTypes = ["make_call"]

This is the mechanism that lets a GHL workflow directly launch a Newo outbound call.

19.3 Native SMS Action​

The SMS native action is implemented using the same overall pattern as the outbound call action:

  • authenticate to Newo using the provided API key
  • resolve or create the correct persona/actors
  • send an external_request into Newo
  • pass the task instruction for SMS execution

From an implementation and operational perspective, users should treat it the same way:

  • configure the required input fields in the GHL action
  • insert the action into the desired GHL workflow
  • let Newo handle the actual execution side

Use cases:

  • send an immediate follow-up SMS after form submission
  • send an SMS after a missed call workflow branch
  • send a callback confirmation message
  • send nurture or reminder content from a pipeline stage change

19.4 How Users Embed Native Actions in GHL Pipelines​

Users can place these native actions inside GoHighLevel workflows and pipeline-driven automations.

Typical pattern:

  1. Create or edit a GHL workflow.
  2. Choose the business trigger, for example:
    • form submitted
    • contact created
    • opportunity stage changed
    • missed call / no answer follow-up branch
  3. Add the native Newo action from the Marketplace app.
  4. Map workflow fields to the action inputs.
  5. Publish the workflow.
  6. Test on a demo contact.

Typical mapping for the outbound call action:

  • contact phone -> newo_phone_number
  • contact full name -> newo_contact_name
  • internal app/API key -> newo_api_key
  • optional workflow-specific instruction -> newo_agent_instruction

Examples of good newo_agent_instruction values:

  • Introduce yourself as the clinic's AI assistant and help the lead schedule a consultation.
  • This lead requested a callback after missing our previous call. Ask for a better callback time if they do not want to speak now.
  • Explain that you are following up on their form submission and qualify their interest before booking.

19.5 Native Action Safety and Governance​

Partner teams should define guardrails before enabling native actions in production.

Recommended safeguards:

  • business hours only
  • limit number of retry attempts
  • do not auto-call the same lead repeatedly in a short window
  • use different instructions for hot leads vs nurture leads
  • always test with a demo contact before enabling for the full pipeline

For SMS:

  • respect quiet hours
  • prevent duplicate sends
  • keep message copy compliant and concise
  • ensure opt-in and compliance logic are handled appropriately on the client side

19.6 Workflow Automations Based on Synced Data​

Depending on feature flags, the integration can write:

  • outbound conversation messages into GHL Conversations
  • call outcome/status information
  • contact notes / conversation history
  • opportunity and pipeline stage changes

This is still useful for GHL-side automations even when native actions are available.

Common patterns:

  • use synced call outcome to branch workflow logic
  • send SMS after voicemail or no-answer
  • create internal callback tasks
  • move leads through nurture logic based on pipeline stage

  • busy
  • no-answer
  • failed

  • voicemail

19.8 Outbound Call Automation from GoHighLevel Workflows​

Typical use case:

  • if the AI agent reached voicemail
  • if the call was not answered
  • if a lead asked for a callback

Recommended logic:

  1. Let the Newo integration finish the conversation and sync the result to GHL.
  2. In GoHighLevel, create a Workflow that listens for the relevant conversation/contact update.
  3. Filter by the synced call outcome, for example:
    • voicemail
    • no-answer
    • failed
  4. Add the GHL action that fits your account setup:
    • launch an outbound call action, if available in that GHL account
    • or create a manual callback task for a team member
  5. Add delays, business-hour windows, and retry rules so the automation does not fire too aggressively.

Important implementation note:

  • the current integration writes the call/session result into GHL
  • the actual outbound call automation is configured and executed inside GoHighLevel
  • availability of specific GHL call actions depends on the client's GHL plan, telephony setup, and enabled features

Recommended GHL-side examples:

  • voicemail -> wait 5 minutes -> trigger callback workflow
  • no-answer -> send SMS first -> create callback task
  • failed -> create internal alert instead of immediate redial

19.9 SMS Automation from GoHighLevel Workflows​

Typical use case:

  • send an SMS after voicemail
  • send an SMS after a missed call
  • send an SMS after a successful booking or follow-up outcome

Recommended logic:

  1. Enable ghl_sync_conversation so session outcomes are written back to GHL.
  2. In GoHighLevel Workflow Builder, create a workflow triggered by the relevant contact/conversation activity.
  3. Add conditions based on the synced result, for example:
    • voicemail
    • no-answer
    • canceled
  4. Add an SMS action in GHL.
  5. Use a short and explicit template, for example:
    • “Sorry we missed you. Reply here if you would like us to call you back.”
    • “We tried to reach you. What time works best for a callback?”
  6. Add guardrails:
    • quiet hours
    • duplicate-prevention rules
    • max retry limits

Important:

  • SMS sending in this pattern is a GHL-side automation
  • the Newo integration does not directly send GHL SMS as part of this specific post-session workflow
  • GHL messaging setup, assigned number, and account permissions must already be configured on the GHL side

Before enabling native actions or GHL-side automations, confirm:

19.12 FAQ for GHL Automations​

Can users trigger outbound calls directly from GoHighLevel?​

Yes. The Marketplace app provides a native action that lets GoHighLevel trigger a Newo outbound call directly.

Can users trigger SMS directly from GoHighLevel?​

Yes. The Marketplace app also supports an SMS action using the same general pattern as the outbound call action.

Can I still use GHL Workflows based on synced call outcomes?​

Yes, if the required session result is synced to GHL and the GHL account is configured for SMS. In this pattern, GHL sends the SMS, not the integration itself.

Which trigger should I use in GoHighLevel?​

For native actions, use the business trigger that should launch the task:

  • form submission
  • opportunity stage change
  • contact creation
  • missed call follow-up branch

For synced-data automations, use the GHL workflow trigger that best matches how the synced data becomes available in that account, typically a contact/conversation update or a workflow condition tied to the synced call result.

What is the safest first automation to enable?​

An SMS follow-up after voicemail or no-answer, or a single controlled outbound callback action for a demo workflow. Both are easier to test than aggressive multi-step retry logic.

Can I use Private Token and still build GHL automations?​

Yes, but behavior may differ depending on token scopes, account entitlements, and channel setup. If workflow behavior is inconsistent, OAuth / Marketplace-style setup is the safer baseline.

What happens inside the native outbound call action?​

It authenticates to Newo using the API key, resolves the correct actor/persona, and sends an external_request with taskTypes = ["make_call"].

19.13 Troubleshooting for Native Actions and GHL Automations​

The native outbound call action does not run​

Check:

  • newo_api_key is present and valid
  • newo_phone_number is populated
  • newo_contact_name is populated
  • the Marketplace action is inserted in a published GHL workflow
  • the workflow branch actually reaches the action

The native action returns phone number is empty​

Check:

  • mapping from GHL contact/workflow field into newo_phone_number
  • whether the contact phone is actually populated
  • whether the phone field is blank after trimming

The native action returns contact name is empty​

Check:

  • mapping from GHL field into newo_contact_name
  • whether the workflow uses a field that is sometimes blank
  • whether first/last name fields need to be combined before mapping

The native action authenticates but does not create the expected task​

Check:

  • Newo API key validity
  • whether actor/persona creation is succeeding
  • whether the external_request path is enabled in the target Newo environment
  • whether the workflow instruction in newo_agent_instruction is valid and useful

The GHL Workflow never starts​

Check:

  • ghl_sync_conversation = True
  • the session actually reached ghl_session_ended
  • the expected contact or conversation update exists in GHL
  • the workflow trigger in GHL matches the object that was updated

The status value is not what we expected​

Check:

  • whether the final synced outcome is one of:
    • completed
    • answered
    • busy
    • no-answer
    • failed
    • canceled
    • voicemail
  • whether the workflow condition uses the same exact value

SMS workflow starts but no message is sent​

Check:

  • GHL SMS capability is configured for that location
  • the contact has a valid phone number
  • quiet hours or compliance rules are not blocking the send
  • the workflow action uses the correct sending number/channel

Outbound call workflow starts but no call happens​

Check:

  • the GHL account/plan supports the specific calling action being used
  • the calling number/channel is configured
  • business hours and retry guards are not preventing execution
  • the workflow is not paused or filtered incorrectly

The workflow fires too often​

Add:

  • deduplication conditions
  • wait steps
  • max retry counters
  • exclusion logic for recent prior attempts

Private Token setup behaves differently from OAuth​

Check:

  • token scopes
  • account entitlements
  • whether the deployment depends on Marketplace-installed app behavior
